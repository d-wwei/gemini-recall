# Gemini CLI 个人助理初始化系统

[English](./README.md)

> **正在寻找其他 CLI 助手的版本？**
> 看看兄弟扩展项目：[Claude Code Companion Starter](https://github.com/d-wwei/claude-code-companion-starter) & [Codex Companion Starter](https://github.com/d-wwei/Codex-companion-starter)

把 Gemini CLI 调整成一个更稳定、更懂工作区上下文、适合长期协作的个人助理部署包。

这是一个面向 Gemini CLI 的轻量部署包，适合那些不满足于“一次性问答”，而希望 Gemini 在文件系统里建立轻量外部记忆层的人。这样做的目的，是让协作在跨会话时依然保持连续，而不是每次重新开始。

## 这个仓库解决什么问题

很多“个人助理型提示词”在理论上看起来很完整，但放到真实 CLI 环境里会出现几个常见问题：

- 过度依赖聊天记录，一刷新就失忆
- 每轮都试图读取过多文件，导致响应变慢、Token 浪费
- 把“文件示例输出”当成“真的写入文件”
- 把长期规则和临时上下文混在一起

这个仓库就是围绕这些现实问题设计的。

它不假设 Gemini 会天然记住一切，而是推动它把文件系统当作长期协作界面的一部分。

## 核心特性快速概览

这个提示词的核心魔力在于：**把 Gemini 从一个每次都要重新磨合的“超强单次工具人”，变成一个能长久记住你、陪你积累默契的“个人专属助理”。**

- **长期记忆体系**：自动建立跨项目的全局记忆（`~/.gemini/`）和项目私有记忆（`.assistant/`）。
- **偏好与习惯积累**：记住你的身份、代码风格、工作推进流、重复的高频指令。
- **自动记忆提炼**：在具体的项目实践中，自动发掘你的习惯和可复用经验，并提供提炼到全局记忆的能力。
- **项目专属定制**：每个项目互不干扰，可以拥有自己独有的上下文结构和覆盖全局的个性化记忆。
- **持续协作不断崖**：下次重新打开项目，它记得你们上次聊到了哪里，不再需要从零开始解释背景。

## 为什么要用 OpenClaw 风格记忆系统改造 Gemini CLI

Gemini CLI 很适合执行指令、读写本地文件，但这并不自动等于它已经具备长期协作系统的能力。

OpenClaw 风格记忆系统的价值，在于把文件视作明确的长期记忆资产，而不是把所有上下文都压在聊天历史里。这种思路特别适合 CLI 助手：

- 稳定偏好应该进入结构化文件
- 项目决策应该跨会话保留
- 临时信息应该和长期记忆分开

这个仓库把这套思路落到了 Gemini CLI 上：用 `.assistant/` 承载项目本地记忆结构，用 `~/.gemini/GEMINI.md` 承载全局默认行为层。

最终目的不是“把 Gemini 变成 OpenClaw”，而是给 Gemini CLI 增加一层更清晰的外部记忆表面，让它少一点无状态工具感，多一点持续协作感。

## 它改变了什么

- 偏好和协作习惯不再只散落在聊天记录里
- 项目上下文变得可检查、可编辑
- 临时信息和长期信息分层存放
- Gemini 重新进入工作区时会更有连续性

## 具体例子

1. 表达风格不再每次重讲。  
   例如“保持简洁、先给结论、再讲风险”，可以进入 `.assistant/STYLE.md`。

2. 项目决策不会轻易重置。  
   例如“先不做大重构”的原因，可以进入 `.assistant/memory/projects/architecture.md`。

3. 每日临时笔记不会污染长期规则。  
   例如未确认信息或当天上下文，可以进入 `.assistant/memory/daily/YYYY-MM-DD.md`。

4. 工作推进方式可以个性化。  
   例如“先检查、再修改、最后汇报验证”，可以进入 `.assistant/WORKFLOW.md`。

## 核心思路

- 用文件承载记忆，而不是只依赖聊天历史
- 用懒加载减少每轮读取成本
- 用增量编辑代替粗暴重写
- 强调真实文件操作，避免“假装已写入”
- 用轻量 bootstrap 代替冗长 onboarding

## 这个提示词的定位

这个仓库目前只提供一个主提示词版本，而不是多版本分支。

可以把它理解为一个相对平衡的工作版本：

- 足够主动，能创建文件和初始化结构
- 又不至于默认大范围覆盖已有内容
- 对记忆结构和懒加载有比较明确的工程立场

## 设计原则

- 把全局配置当成默认行为层，而不是永久强控制层
- 用项目本地记忆承载持续演化的上下文
- 不让临时信息污染长期记忆
- 小而稳定的文件，比巨型 prompt 更适合长期维护
- 真实 CLI 行为边界，比理想化代理设想更重要

## 重要边界

- `~/.gemini/GEMINI.md` 更适合被理解为默认行为层，不是永久强制控制层。
- `.assistant/` 是项目本地记忆约定，不是 Gemini CLI 天然内建协议。
- 最终效果仍然取决于运行时上下文、可用工具和模型本身的执行行为。

## FAQ

### 为什么不每次都直接发一个超长总提示词？

因为超长提示词不适合长期协作。

一旦任务开始跨会话持续，你就需要稳定地保存这些内容：

- 用户偏好
- 项目决策
- 临时上下文
- 未完成事项

这些东西放在文件里，比反复在聊天里重讲更稳。

### 为什么不把所有内容都放进 `GEMINI.md`？

因为全局规则和项目记忆不是一回事。

`GEMINI.md` 更适合承载默认行为和稳定规则；项目目录下的文件更适合保存会随着实际工作逐渐变化的本地上下文。

### `.assistant/` 会不会太重？

会，如果设计得太复杂。

所以这个仓库尽量把结构控制在一个小而实用的范围内。目标不是增加流程负担，而是减少重复磨合。

### 这是不是想取代 Gemini CLI？

不是。

它只是提示词层面的工作流增强。真正执行任务的仍然是 Gemini CLI，这个仓库做的是给它补一层更耐用的协作结构。

## 适合谁用

如果你希望 Gemini CLI 更像下面这些角色，这个仓库会比较有用：

- 可重复使用的项目助理
- 对工作区有感知的协作者
- 能跨会话维护轻量上下文的工具

如果你只需要偶尔发一个一次性 prompt，不关心本地记忆结构，那它的价值就会小很多。

## 备注

如果 Gemini 在你的环境里显得过于激进，可以收紧提示词里的执行约束或缩小初始化范围。

如果它过于保守、推进不足，就加强文件创建、校验和 bootstrap 跟进要求。
